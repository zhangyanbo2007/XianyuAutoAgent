真实世界的硬件项目需要对异构工件进行推理，包括RTL源代码、验证组件、IP配置和构建脚本，以及与项目原生构建和仿真流程的交互。因此，一个基本问题仍然悬而未决：基于LLM的智能体能否在真实硬件项目的完整上下文中执行存储库级别的错误修复？存储库级别的、基于执行的评估推动了软件工程的实质性进展，其中SWE-bench [8]及其扩展 [6, 15] 等基准测试实现了对智能体能力的标准化、可重现评估。硬件领域尚不存在可比的基础设施，并且构建它带来了独特的挑战。开源HDL存储库稀缺：在标准代码语料库中，Verilog和SystemVerilog的文件数量比Python少100-1,000倍 [13]，更不用说Chisel等硬件DSL了。即使识别出合适的存储库，构建可重现、可验证的基准实例仍然很困难：软件基准可以依赖标准化的构建和测试基础设施，而硬件验证则依赖于异构的、通常是专有的工具链，并且许多错误缺乏自动化回归测试。因此，为每个任务实例建立可重现的验证需要大量的体力劳动。为了弥补这一空白，我们引入了HWE-Bench，这是第一个大规模、存储库级别的基准测试，用于评估LLM智能体在真实世界硬件错误修复任务中的表现。如图1所示，每个任务实例为LLM智能体提供一个错误报告和完整的硬件存储库；智能体通过文件读取、编辑和shell执行等工具与代码库交互，生成一个补丁，然后通过项目原生的仿真套件在容器化环境中进行验证。HWE-Bench包含从六个主要开源项目（OpenTitan、Caliptra、香山、Ibex、CVA6和Rocket Chip）中精选的417个任务实例，涵盖传统HDL（Verilog/SystemVerilog）和硬件DSL（Chisel），设计类别多样，从RISCV核心到片上系统平台和安全信任根。每个任务都基于一个完全容器化的、基于执行的评估环境。该基准测试通过一个系统化的、大部分自动化的管道构建，该管道从存储库噪声中过滤出真正的硬件错误修复，并为每个实例建立可重现的容器化验证。我们使用四个智能体框架，在所有六个存储库上评估了七个LLM，包括专有和开源模型。表现最佳的智能体实现了70.7%的总体解决率，但现有的硬件设计基准主要评估大型语言模型（LLMs）在孤立的、组件级任务（例如从规范生成HDL模块）上的表现，而存储库规模的评估尚未解决。我们引入了HWE-Bench，这是第一个大规模、存储库级别的基准测试，用于评估LLM智能体在真实世界硬件错误修复任务中的表现。HWE-Bench包含

原始摘要

real-world hardware projects demand reasoning across heterogeneous artifacts including RTL source, verification components, IP configurations, and build scripts, as well as interaction with projectnative build and simulation flows. A fundamental question therefore remains open: can LLM-based agents perform repository-level bug repair within the full context of a real hardware project? Repository-level, execution-grounded evaluation has driven substantial progress in software engineering, where benchmarks such as SWE-bench [8] and its extensions [6, 15] enabled standardized, reproducible evaluation of agent capabilities. No comparable infrastructure exists for hardware, and building it poses distinct challenges. Open-source HDL repositories are scarce: Verilog and SystemVerilog comprise 100–1,000× fewer files than Python in standard code corpora [13], let alone hardware DSLs such as Chisel. Even when suitable repositories are identified, constructing reproducible, verifiable benchmark instances remains difficult: where software benchmarks can rely on standardized build and test infrastructure, hardware verification depends on heterogeneous, often proprietary toolchains, and many bugs lack automated regression tests. Establishing reproducible validation for each task instance therefore demands substantial manual effort. To address this gap, we introduce HWE-Bench, the first largescale, repository-level benchmark for evaluating LLM agents on real-world hardware bug repair tasks. As illustrated in Figure 1, each task instance provides an LLM agent with a bug report and the full hardware repository; the agent interacts with the codebase through tools such as file reading, editing, and shell execution to produce a patch, which is then validated through the project’s native simulation suite in a containerized environment. HWE-Bench comprises 417 task instances curated from six major open-source projects (OpenTitan, Caliptra, XiangShan, Ibex, CVA6, and Rocket Chip), spanning both traditional HDLs (Verilog/SystemVerilog) and a hardware DSL (Chisel), with diverse design categories from RISCV cores to system-on-chip platforms and security roots-of-trust. Each task is grounded in a fully containerized, execution-based evaluation environment. The benchmark is constructed through a systematic, largely automated pipeline that filters genuine hardware bug fixes from repository noise and establishes reproducible, containerized validation for each instance. We evaluate seven LLMs, both proprietary and open-source, using four agent frameworks across all six repositories.
