原始摘要与中文对照

中文对照翻译

真实世界环境。为了克服这一限制，赋予智能体自我演化能力——使其能够根据环境反馈自动调整策略、完善指令并更新工具——已成为实现鲁棒自主性的关键途径。这种从预定义执行到动态适应的转变代表了智能体系统设计的根本性转变。基于LLM的智能体系统在处理复杂、长周期任务方面展现出潜力。然而，现有的智能体协议（例如A2A和MCP）对跨实体生命周期和上下文管理、版本跟踪以及演化安全更新接口的规定不足，这导致了单体组合和脆弱的胶水代码。我们引入了AUTOGENESIS P RO TOCOL (AGP)，一种将“演化什么”与“如何演化”解耦的自我演化协议。其资源基底协议层（RSPL）将提示、智能体、工具、环境和内存建模为具有明确状态、生命周期和版本化接口的协议注册资源1。其自我演化协议层（SEPL）指定了一个闭环操作员接口，用于提议、评估和提交具有可审计沿袭和回滚能力的改进。基于AGP，我们提出了AUTOGENESIS S YSTEM (AGS)，一个在执行过程中动态实例化、检索和完善协议注册资源的自我演化多智能体系统。我们在多个需要长周期规划和跨异构资源工具使用的挑战性基准上评估了AGS。结果表明，与强大的基线相比，AGS持续改进，支持了智能体资源管理和闭环自我演化的有效性。尽管对自我演化智能体的兴趣日益增长，但其实现仍然高度碎片化和临时性。现有系统通常缺乏共享标准，使得演化过程既不可组合也无法审计。开发者经常被迫依赖脆弱的胶水代码，导致难以维护的单体架构。此外，如果没有明确的生命周期管理和安全更新接口，自我修改会引入显著的运行时不稳定风险。为了解决这些问题，有必要将开发从临时工程实践提升到协议层面，通过标准化框架将“演化什么”与“如何演化”解耦，以确保模块化、可追溯和安全的演化。虽然Anthropic的Model Context Protocol (MCP) (Anthropic, 2025b)和Google的Agent-toAgent (A2A)等协议已经标准化了连接性，但将它们直接应用于自我演化场景存在概念上的不匹配。这些协议主要旨在解决连接性挑战——具体来说，是模型-工具调用（MCP）或智能体间通信（A2A）。然而，自我演化的核心不在于调用，而在于状态突变和管理。现有连接协议缺乏对实体生命周期和版本沿袭的原生支持。在闭环演化系统中，如果组件的创建、更新和销毁没有精确定义，优化器就无法安全地应用修改。此外，缺乏版本跟踪和回滚机制意味着错误的更新可能导致不可恢复的错误。因此，仅仅依靠通信协议是不够的；需要一种能够管理突变动态的新型协议。

原始摘要

real-world environments. To overcome this limitation, endowing agents with self-evolution capabilities—enabling them to automatically adjust strategies, refine instructions, and update tools based on environmental feedback—has emerged as a critical avenue for achieving robust autonomy. This transition from predefined execution to dynamic adaptation represents a fundamental shift in agentic system design. Recent advances in LLM based agent systems have shown promise in tackling complex, long horizon tasks. However, existing agent protocols (e.g., A2A and MCP) under specify cross entity lifecycle and context management, version tracking, and evolution safe update interfaces, which encourages monolithic compositions and brittle glue code. We introduce AUTOGENESIS P RO TOCOL (AGP), a self evolution protocol that decouples what evolves from how evolution occurs. Its Resource Substrate Protocol Layer (RSPL) models prompts, agents, tools, environments, and memory as protocol registered resources1 with explicit state, lifecycle, and versioned interfaces. Its Self Evolution Protocol Layer (SEPL) specifies a closed loop operator interface for proposing, assessing, and committing improvements with auditable lineage and rollback. Building on AGP, we present AUTOGENESIS S YSTEM (AGS), a self-evolving multi-agent system that dynamically instantiates, retrieves, and refines protocol-registered resources during execution. We evaluate AGS on multiple challenging benchmarks that require long horizon planning and tool use across heterogeneous resources. The results demonstrate consistent improvements over strong baselines, supporting the effectiveness of agent resource management and closed loop self evolution. Despite the growing interest in self-evolving agents, implementations remain largely fragmented and ad hoc. Existing systems often lack shared standards, rendering the evolution process neither composable nor auditable. Developers are frequently forced to rely on brittle glue code, leading to monolithic architectures that are difficult to maintain. Furthermore, without explicit lifecycle management and safe update interfaces, self-modification introduces significant risks of runtime instability. To address these issues, it is necessary to elevate development from ad hoc engineering practices to the protocol level, decoupling “what evolves” from “how evolution occurs” via a standardized framework to ensure modular, traceable, and safe evolution. While protocols such as Anthropic’s Model Context Protocol (MCP) (Anthropic, 2025b) and Google’s Agent-toAgent (A2A) have standardized connectivity, applying them directly to self-evolution scenarios presents a conceptual mismatch. These protocols are primarily designed to resolve connectivity challenges—specifically, model-tool invocation (MCP) or inter-agent communication (A2A). However, the core of self-evolution lies not in invocation, but in state mutation and management. Existing connectivity protocols lack native support for entity Lifecycle and Version Lineage. In a closed-loop evolutionary system, if the creation, update, and destruction of components are not precisely defined, the optimizer cannot safely apply modifications. Moreover, the absence of version tracking and rollback mechanisms means that erroneous updates can lead to irrecoverable errors. Consequently, relying solely on communication protocols is insufficient; a novel protocol capable of managing the dynamics of mutation is required.
